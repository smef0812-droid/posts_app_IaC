# posts_app_IaC

## 概要
Laravelで作成した投稿アプリケーションを、AWS上に三層アーキテクチャで構築し、TerraformでIaC化したプロジェクトです。

インフラ構成の理解と、クラウド設計力の向上を目的として作成しました。

---

## アーキテクチャ構成

- ALB（Application Load Balancer）
- EC2（アプリケーションサーバー）
- RDS（データベース）
- （Terraformで構成管理）

三層構造を採用し、可用性・セキュリティを意識した設計としています。

---

## 設計意図

### ■ セキュリティ
- RDSは外部から直接アクセスできないよう、プライベートサブネットに配置
- EC2もプライベートサブネットに配置し、ALB経由のみアクセス可能
- セキュリティグループにより通信経路を制限

### ■ 可用性
- ALBを前段に配置し、負荷分散を実現
- 将来的なAuto Scaling構成への拡張を想定

### ■ 柔軟性・拡張性
- ALB構成とすることで、EC2からECSなどへの移行が容易な設計
- コンピューティング層を抽象化し、将来的な構成変更に対応可能

### ■ 通信設計
- EC2から外部リソース（例：GitHub）へのアクセスを考慮し、NAT Gateway導入を想定

---

## 技術選定理由

本構成ではEC2を採用していますが、これは学習目的によるものです。

実際の運用においては、以下のような選択肢が適切と考えています：

- 小規模・低トラフィック  
  → API Gateway + Lambda（サーバーレス構成）

- 中〜大規模・可用性重視  
  → ECS（コンテナベース）

用途やトラフィックに応じて最適な構成を選択することが重要と考えています。

---

## 現状の課題と改善案

### ■ ① HTTPS未対応（セキュリティ課題）
現状はHTTP通信のため、安全性に課題あり。

**改善案**
- ACM（AWS Certificate Manager）で証明書発行
- ALBでHTTPS終端

---

### ■ ② NAT Gateway未導入
外部通信が制限されており、以下の問題あり：
- パッケージ更新不可
- セキュリティアップデート不可

**改善案**
- NAT Gatewayを導入し、安全なアウトバウンド通信を確保

---

### ■ ③ セッション管理の問題
セッション情報をEC2に保持しているため、  
ALBによる負荷分散時に以下の問題が発生：

- 別インスタンスにルーティングされた場合、ログイン状態が維持されない

**改善案**
- ElastiCache（Redis）でセッション共有
- またはALBのスティッキーセッションを利用
